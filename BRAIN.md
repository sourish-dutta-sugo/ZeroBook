## 1. What this project actually is (ground truth vs. aspiration)

| Claim source | What it says |
|---|---|
| **This repo, right now** | A 100%-native Android app. `app/build.gradle.kts` → `applicationId = "com.aistudio.retailbook.acntgp"`, `namespace = "com.example"`, `versionName = "2.2.1"`, `versionCode = 4`. No SQLDelight, no KMP module, no `shared/` source set anywhere in the tree. |
| **README.md (in repo)** | States this is "the original Android-only codebase" and that ZeroBook is "currently being rebuilt as Kotlin Multiplatform... using SQLDelight in place of Room." That rebuild is **not in this repository** — it's described only as a roadmap note. |
| **Out-of-band context (not in repo)** | Other notes referencing a `com.zerobook.app` package, a partially-migrated KMP scaffold (28/52 files in `commonMain`), and a Cloudflare Workers + Resend email backend exist as *separate, unpushed work*. **None of that code is present here.** If you are an agent picking up "ZeroBook" work and you see references to those things, you are looking at a different checkout or a future state — do not assume this repo contains them. |

**Bottom line:** treat this repository as the Room/Compose/WorkManager/JavaMail Android app described below. Anything about Cloudflare Workers, Resend, SQLCipher, or KMP is *planned*, not *implemented*, in this snapshot.

### Product framing (why this exists)
ZeroBook ("Record. Transact. Grow.") targets Indian small retailers (kirana stores, small traders) who are too small for Tally Prime but have outgrown pen-and-paper ledgers. The product wedge is: GST-compliant invoicing + double-entry-style ledgers + inventory + reports, in one mobile-first app, usable one-handed in a shop, without an accountant. License (`README.md` bottom half) is a custom "view/fork/contribute but don't rebrand or republish under another name" license — relevant if anyone proposes whitelabeling.

## 4. Navigation / screen map

`MainActivity.kt` hosts a single `NavHost` (`Routes` object, line ~108) with `startDestination = Routes.Dashboard`. Routes:

```
dashboard            → DashboardScreen      (home, universal search, quick actions)
vouchers             → VouchersScreen        (list + filter all voucher types)
parties              → PartiesScreen         (customer/supplier list + ledgers)
settings             → SettingsScreen        (business profile, theme, SMTP, reminders, FY mgmt)
reports              → ReportsScreen         (business + stock reports)
ledger_books         → LedgerListScreen      (chart of accounts / ledger drill-down)
expenses             → ExpensesScreen
quick_sale           → QuickSaleScreen       (fast counter-sale flow)
products             → ProductsScreen        (inventory/catalog)
bank_cash            → BankCashScreen
new_voucher?voucherId={voucherId} → VouchersScreen edit/create (voucherId optional → new)
invoice/{voucherId}  → InvoiceScreen         (PDF preview/share/print/email after posting)
party_detail/{partyId} → PartySheets-driven detail view
```

**App-launch decision tree** (`MainActivity.kt` ~line 272–310, backed by `AppViewModel.isSetupCompleted`):
1. `installSplashScreen()` (Android 12 splash API) + custom `SplashScreen` composable for branded splash. Splash text must read **"ZeroBook"** (confirmed in manifest `android:label`), not any earlier project name.
2. `AppViewModel` loads `BusinessProfile` via `getProfileSync()`; `isSetupCompleted.value = (profile != null)`.
3. If **not** setup → SetupScreen (first-run onboarding: business name/address/GST/PAN/bank, with PIN-code-based city/state autofill — implemented in `SetupScreen.kt` + `ui/screens/ProfileFormSupport.kt`).
4. If setup is complete but a PIN lock is configured (`pinRequired`) and not yet authenticated this session (`pinAuthed`) → PIN entry gate.
5. Otherwise → Dashboard via the real `NavHost` (back stack semantics: top-level tabs use state-preserving navigation, not stack-clearing; voucher-save → invoice screen is an intentional **replace**, not a push, so back-from-invoice doesn't return to the edit form).

**Bottom nav** = `topLevelDestinations`: Dashboard, Vouchers, Parties, Settings (icons only — Reports/Products/BankCash/Expenses are reached from within Dashboard/Settings shortcuts, not the bottom bar).

If you add a new top-level screen: add to `Routes`, add a `composable(...)` block in `ZeroBookNavHost`, and decide whether it belongs in `topLevelDestinations` (bottom nav) or is reached contextually. Forgetting the `navigateToTopLevel` state-preservation helper will cause tab switches to lose scroll/filter state.

## 5. Data model (Room schema, DB version 13)

All entities live in `Entities.kt`; DAOs in `AppDaos.kt`; the `@Database` annotation in `AppDatabase.kt` is the authoritative entity list. Primary keys are mostly `String` UUIDs (parties, products, vouchers, voucher items, ledger entries, bank/cash transactions, ledger accounts, bills receivable, expenses) — chosen so client-generated IDs never collide and so future multi-device sync wouldn't require ID renegotiation. `financial_years.code` is a natural key (`"2025-26"` style string), not a UUID.

### Core entities and their role
- **`FinancialYear`** — one row per FY (`code`, `startDate`, `endDate`, `isClosed`, `isLocked`, `sourceFinancialYearCode` for traceability of year-close lineage).
- **`BusinessProfile`** — singleton row (`id = 1`, enforced by app logic, not a DB constraint). Holds GSTIN/PAN/bank/UPI, invoice display toggles, **and SMTP credentials in plaintext** (`smtp_email`, `smtp_password`, `smtp_host`, `smtp_port`).
- **`Party`** — customers/suppliers (`type`: CUSTOMER/SUPPLIER/BOTH), opening balance + balance type (DR/CR), credit terms, plus denormalized analytics columns (`total_purchases_amount`, `total_transactions`, `first/last_transaction_date`, `loyalty_points`) that are **recomputed**, not authoritative — see `updatePartyAnalytics()`.
- **`Product`** — catalog with GST rate, HSN, stock fields. `current_stock` is a **derived/cached** column recomputed by `recalculateProductStocks()`; never trust it as the source of truth if vouchers were edited outside the repository's transactional path.
- **`Voucher`** — the universal transaction header. `type` is a string enum (`SALE`, `PURCHASE`, `SALE_RETURN`, `PURCHASE_RETURN`, `RECEIPT`, `PAYMENT`, `DEBIT_NOTE`, `CREDIT_NOTE`, `JOURNAL`, `BILLS_RECEIVABLE`, `BILLS_PAYABLE`, `QUOTATION`, `DELIVERY_CHALLAN`). `status` is `DRAFT`/`POSTED` — **only `POSTED` vouchers generate ledger entries / stock movement / bank-cash transactions** (`shouldPostAccounts` gate in `saveAndPostVoucher`). GST split fields (`cgst`/`sgst`/`igst`) plus `isIgst` flag are pre-computed at save time, not derived at read time.
- **`VoucherItem`** — line items, GST computed per-line and also rolled up to the voucher header (denormalized in two places — must stay consistent; see §8).
- **`LedgerEntry`** — the actual double-entry rows (`accountHead` is a free-text string, not a foreign key to `LedgerAccount`! See §8 risk).
- **`BankCashTransaction`**, **`ReceiptAllocation`**, **`LedgerAccount`**, **`BillReceivable`** — supporting bookkeeping tables.
- **`*FinancialYearBalance`** tables (Party/Product/LedgerAccount) — opening-balance snapshots **per financial year**, populated by `closeFinancialYear()`. This is how the app avoids re-summing the entire transaction history back to year zero every time it needs an opening balance.
- **`FinancialYearAuditLog`** — append-only log of year-close events (idempotency guard: checks for an existing `YEAR_CLOSE_COMPLETED` log before re-logging).
- **`Expense`** — separate from vouchers entirely; not part of the double-entry ledger flow.

### Migration & schema-evolution convention (important, somewhat unusual)
The project uses **two layers of schema evolution simultaneously**:
1. Formal Room `Migration` objects (`MIGRATION_4_5` … `MIGRATION_12_13` in `Migrations.kt`) run once when `versionCode` bumps.
2. **Idempotent `ensureColumn(...)` / `ensure*Table(...)` calls** wired into `RoomDatabase.Callback.onCreate` **and** `onOpen` in `AppDatabase.kt`. These run on *every single app open*, checking `PRAGMA table_info` (implied by "ensureColumn" naming) before issuing `ALTER TABLE ... ADD COLUMN` if missing.

**Why both exist:** the defensive `ensure*` callbacks were almost certainly added because the formal migration chain fell out of sync with the entity classes at some point (a classic symptom of an AI-assisted, screenshot-driven workflow where new columns get added to `Entities.kt` faster than a matching `Migration` gets written). The `onOpen` guard is a safety net that makes the app self-healing even if a migration was skipped — but it also means **schema correctness is partially decoupled from `exportSchema`/migration testing**, so Room's own migration test tooling won't catch every drift.

**Rule for future schema changes:** Add the column to the entity AND to a real `Migration` AND to the matching `ensureColumn` guard. Skipping the guard is "fine" for fresh installs but will leave existing users' DBs missing the column until they hit whatever code path indirectly calls the guard again — currently every app open, so in practice this is low-risk *for this app's current callback wiring*, but if that wiring is ever refactored, silently-missing columns become a real crash risk (`no such column` SQLiteException) for upgraded users specifically, not new installs.

### What breaks if you touch this layer
- Removing a field from an `@Entity` without a migration that drops/renames the column will not break Room (it just stops reading/writing that column) but **will leave orphaned data** in old installs' SQLite files.
- Changing a `@PrimaryKey` type or a table name requires a real destructive-or-handled migration; the `ensureColumn` pattern only adds columns, it never renames/drops, so don't rely on it for anything beyond additive changes.
- `exportSchema = false` means there's no `schemas/` JSON history checked in — Room's migration-test framework (`MigrationTestHelper`) can't be used as-is; if test coverage is ever added here, flip this to `true` and start checking in schema JSONs first.

## 6. Core business logic & workflows

### 6.1 GST calculation (place-of-supply detection)
Interstate vs. intrastate is decided client-side, per voucher, by comparing the selling business's `BusinessProfile.stateCode` to the party's `Party.stateCode`:
```kotlin
val isInterstate = party.stateCode.isNotEmpty() && party.stateCode != profile.stateCode
```
If interstate → IGST only (full GST rate on one line). If intrastate → CGST + SGST, each exactly half the GST rate. This logic is **duplicated inline** in `VouchersScreen.kt` (the main voucher editor) and reproduced with slightly different variable names in `QuickSaleScreen.kt` / `ReportsScreen.kt` (manual journal-style entries default to `isIgst = false`, i.e., they assume intrastate and don't run the comparison at all). **Risk:** if the interstate-detection formula ever needs a tweak (e.g., handling a blank/unregistered party GSTIN, or a union-territory edge case), it must be changed in every screen that creates a `Voucher`, not in one shared function. There is no single `GstCalculator` object — this is the single highest-value refactor target if GST logic ever needs to change (see §8).

### 6.2 Voucher save → posting pipeline (`AppRepository.saveAndPostVoucher`)
This is the single most important function in the app — almost every user action funnels through it. Sequence, inside one `db.withTransaction`:
1. Resolve the voucher's financial year (`FinancialYearUtils.financialYearCodeFor(voucher.date)` if not explicitly set) and **fail fast** (`require`) if that FY is locked.
2. **Delete-then-reinsert** pattern: any existing voucher with the same ID, plus its items, ledger entries, bank/cash transactions, receipt allocations, and bills-receivable row, are deleted first — `saveAndPostVoucher` is simultaneously the "create" and "edit" path. There is no separate update path; editing a posted voucher fully regenerates all downstream rows.
3. Insert the voucher header + line items.
4. **If and only if `status == "POSTED"` and type is not `QUOTATION`/`DELIVERY_CHALLAN`**, generate the double-entry `LedgerEntry` rows. The accounting rules are hardcoded per voucher type in a big `when` block (SALE, PURCHASE, BILLS_RECEIVABLE, BILLS_PAYABLE — note `SALE_RETURN`/`PURCHASE_RETURN`/`RECEIPT`/`PAYMENT`/notes are handled further down in the same function, truncated in this summary but follow the same per-type pattern).
   - SALE: handles three payment-mode branches — full CASH/BANK (single DR Cash-or-Bank, CR Sales+GST), CREDIT (DR Party), and the more complex **PART PAYMENT** sub-mode (splits into a paid portion to Cash/Bank and a remaining-credit portion to the Party account) using fields from `VoucherSaveExtras`.
   - GST liability lines (`CGST Payable`/`SGST Payable`/`IGST Payable` for sales, `...Receivable` for purchases) are only added if the respective amount is `> 0`.
   - A `Round Off Account` line is added only if `voucher.roundOff != 0.0`.
5. Persist `VoucherSaveExtras` (partial payment metadata, credit due date, advance flags, reference numbers) via a **raw `execSQL` UPDATE**, not through a typed DAO method (`saveVoucherExtras` — bypasses Room's normal insert/update path; works but means these specific columns aren't covered by Room's compile-time query validation).
6. Auto-create a `BankCashTransaction` for RECEIPT/SALE/PURCHASE_RETURN (treated as money-in) or PAYMENT/SALE_RETURN (money-out), **unless** payment mode is `CREDIT`.
7. If it's a credit (or partial-credit) SALE, auto-create a `BillReceivable` row with a due date — either parsed from `extras.creditDueDate` or defaulted to **`voucher.date + 15 days`** (hardcoded 15-day credit term default — a business rule worth flagging to the product owner if it's not actually configurable per-party despite `Party.creditDays` existing as a field).
8. Finally, **three full recalculation passes run on every save**: `recalculateOutstandings()`, `recalculateProductStocks(fy)`, `updatePartyAnalytics(fy)`. These re-derive denormalized state from scratch for the *entire* financial year's data, not incrementally. See §8 performance note.

### 6.3 Financial year close (`closeFinancialYear`)
Carries forward opening balances into a target FY (default: `FinancialYearUtils.nextFinancialYear(source)`):
- Computes closing ledger-account balances by walking **every** ledger entry for the source year and netting against opening balance (signed by DR/CR convention via `signedAmount`).
- Computes closing party balances by filtering ledger entries whose `accountHead == "Party: ${party.name}"` — **this is a string-match join, not a foreign key** (see §8 risk: renaming a party breaks historical ledger attribution).
- Computes closing product stock by replaying all voucher-item movements for the year, with stock floor at zero (`coerceAtLeast(0.0)`) — meaning the system cannot represent negative stock even transiently, which could silently mask a data-entry-order bug (e.g., recording a sale before its corresponding purchase).
- Writes one `FinancialYearAuditLog` row guarded by idempotency check (`getLog(...)` lookup before insert) so re-running the close for the same year pair doesn't double-log — but note this guard does **not** prevent re-running the actual balance-carry-forward logic itself, only the audit log insert; closing the same year twice would recompute (and overwrite, via `upsertBalances`' REPLACE conflict strategy) the target year's opening balances, which is probably fine (idempotent overwrite) but has not been explicitly tested against partial-failure scenarios.
- Sets `isClosed = true` always; `isLocked` only if caller passes `lockSourceYear = true`. **A locked year cannot have new vouchers posted to it** (`saveAndPostVoucher`'s `require(yearState?.isLocked != true)` guard) — but a *closed-but-unlocked* year apparently still accepts new vouchers, which would then require re-running the close to keep next-year opening balances correct. There is no automatic re-close trigger; this is a manual-process risk for the end user / product owner to be aware of.

### 6.4 Email reminders — two distinct delivery mechanisms (don't conflate them)
1. **`processReminders()`** (driven by `ReminderWorker`/`DueReminderWorker` via WorkManager, and also callable on-demand) — for each party with outstanding `BillReceivable` rows, builds a templated body and calls `launchEmailDraft()`, which fires an `Intent.ACTION_SEND` chooser. **This requires the user to be present to tap "send" in whatever email app they choose** — it cannot truly run unattended in the background despite being scheduled via WorkManager. This is a meaningful UX gap: a "scheduled reminder" silently does nothing useful if the device isn't unlocked/foregrounded when the worker fires, because Android intent-chooser launches from background-running Workers are unreliable/blocked on modern Android versions.
2. **`sendNowForParty()` / `sendTestEmail()`** — these use **`EmailComposer`/raw JavaMail SMTP** (`sendViaSmtp`, reading `BusinessProfile.smtpEmail/smtpPassword/smtpHost/smtpPort`) to send a real email with a PDF invoice attachment, no user interaction required. This is the only path that can truly "send" autonomously, and it's the one most exposed by the security audit (cleartext SMTP password in DB).
   - **If asked to "fix" or "redesign" email reminders, clarify which of these two paths is in scope** — they have different reliability characteristics, different security exposure, and (per the user's stated direction) the long-term plan is to replace path 2 with a Cloudflare Workers + Resend backend, which would eliminate on-device SMTP credentials entirely. That backend does not exist in this repo yet.
3. Reminder *configuration* persistence (`PartyReminderConfig`, templates, selected recipients, logs, schedules) all lives in raw `SharedPreferences("zerobook_pref")`, **separately** from the Room-backed `reminder_schedules` SQL table that `ensureReminderScheduleTable` creates and that `upsertSchedule`/`updateScheduleLastSent` write to via raw `execSQL`. There are effectively **two overlapping reminder-schedule storage systems** (SharedPreferences key/value vs. a real SQL table) — confirm which one any given UI screen actually reads before changing either.

### 6.5 Invoice generation & sharing (`InvoiceGenerator.kt`)
Builds an HTML representation of the invoice (`buildInvoiceHtml`), renders it to PDF via an Android `WebView` print/PDF pipeline (`WebViewPdfWriter.java` + `generatePdfFromVoucherId`), and exposes share/print/export/email entry points. Also generates **UPI deep links** (`generateUpiLink`) for payment QR codes, and computes Indian-style "amount in words" (`amountInWords`) for the printed invoice — a common Indian-invoicing legal/format expectation. `primeVoucherRenderExtras` pre-fetches anything the HTML template needs before rendering — if invoice template fields are ever added, this is the function to extend so the render step has the data it needs without extra round-trips.

### 6.6 Indian-specific conventions baked into the domain model
- Financial year = **April 1 → March 31** (`FinancialYearUtils`), not calendar year — this affects every report, every opening-balance carry-forward, and is the partition key for almost every Room query (`financialYearCode` column on six different tables).
- GST split rules (CGST+SGST for intrastate, IGST for interstate) per §6.1.
- HSN code lookup (`HsnLookup.kt`, duplicated under both `data/` and `utils/` — see §8) for auto-suggesting GST rate from product HSN classification.
- Indian currency formatting (`Utils.formatIndianCurrency`) — almost certainly using the Indian digit-grouping convention (lakh/crore, e.g. `₹1,00,000` not `₹100,000`); don't "fix" this formatting to a Western grouping without realizing it's intentional.
- PIN-code-based address autofill in onboarding/setup (mentioned in product context; implemented somewhere in `SetupScreen.kt`/`ProfileFormSupport.kt`) — uses PIN to derive city/district/state, reducing manual entry burden for a non-technical retailer user base.

## 7. Security posture (as of this snapshot)

This matches and confirms a previously-identified security audit's findings, with direct code evidence:

1. **Unencrypted Room/SQLite database** — `AppDatabase.kt` calls plain `Room.databaseBuilder(...)` with no `SupportFactory`/SQLCipher wrapper. The DB contains GSTIN, PAN, full bank account + IFSC, customer/supplier PII, and (critically) the business's own SMTP password. Anyone with root/backup access to the device (or an `adb backup` if `allowBackup="true"`, which it is — see manifest) can read all of this in plaintext.
2. **Cleartext SMTP credential storage** — `BusinessProfile.smtpPassword` is a plain `String` column, read/written with no encryption, and used directly in `sendViaSmtp`/JavaMail.
3. **`android:allowBackup="true"`** in the manifest, combined with #1, means a full unencrypted financial database can leave the device via Android's backup mechanism unless `dataExtractionRules`/`backup_rules` XML (referenced in the manifest but not inspected line-by-line in this pass) explicitly excludes the database — **verify `xml/backup_rules.xml` and `xml/data_extraction_rules.xml` exclude the Room DB file** before treating backup as safe.
4. **Public/external storage export paths** (`ExportStorageManager.kt`, `CsvTransferManager.kt`) — flagged by the prior audit as writing unencrypted DB/CSV exports to public device storage. `WRITE_EXTERNAL_STORAGE` is requested with `maxSdkVersion="28"` (correctly scoped — modern Android scoped storage applies above API 28), but exported files placed in shared storage on any API level are readable by other apps with storage permission and are not removed when the app is uninstalled.
5. **No certificate pinning / TLS hardening** visible for the SMTP connection — relies on whatever defaults `javax.mail`'s `Properties`-based session config uses (TLS port 587 default is a reasonable baseline, but worth an explicit `mail.smtp.ssl.checkserveridentity=true` style hardening pass if this path is kept).
6. **`smtpPassword` is almost certainly meant to be an app-specific password** (Gmail-style), not the user's main account password, given the default host is `smtp.gmail.com` — but nothing in the UI/code enforces or even explains this distinction to the end user, which is a support/security-education gap as much as a code gap.

**If asked to remediate:** the previously-scoped plan (SQLCipher for the DB, Android Keystore-backed encryption for the SMTP password specifically, backup-rule hardening, and migrating exports off public storage into scoped/app-specific or SAF-mediated storage) is the right shape — none of it is implemented in this snapshot, so any "audit fix" prompt should assume a clean slate here, not partial completion.

## 8. Technical debt, conventions, and risk inventory

Ordered roughly by blast radius if ignored.

1. **Leftover package/namespace from a project rename.** `app/src/main/java/com/vibecoding/zerobook_androidonly0/` still contains a `MainActivity.kt`, `Color.kt`, `Type.kt`, `Theme.kt` — clearly an earlier project name ("vibecoding"/"zerobook_androidonly0") that was renamed to the current `com.example` namespace but never fully deleted. **These files are very likely dead code** (the real `MainActivity` lives at `com/example/MainActivity.kt` and the manifest points there). Before deleting, grep for any remaining imports of the `vibecoding` package to confirm zero references, then remove — this is pure clutter that increases an agent's search space for no benefit.
2. **Duplicate utility/theme implementations.** At least two pairs of near-duplicate files exist:
   - `com/example/data/HsnLookup.kt` (163 lines) vs `com/example/utils/HsnLookup.kt` (148 lines)
   - `com/example/ui/theme/Color.kt` (12 lines) vs the vibecoding package's `Color.kt` (10 lines), plus `com/example/ui/theme/Theme.kt`/`Type.kt` vs their vibecoding counterparts.
   Any agent asked to "fix the HSN lookup" must check **which one is actually imported by the screens that call it** (grep for `import com.example.*.HsnLookup`) before editing — editing the unused copy is a wasted round-trip.
3. **No single GST-calculation function.** Interstate/intrastate + CGST/SGST/IGST split logic is reimplemented per-screen (§6.1). High risk if GST rules ever need a correction (e.g., a special-category-state rule, or composition-scheme handling) since it must be hunted down and fixed in multiple files identically.
4. **`AppRepository.kt` (1733 lines) and `VouchersScreen.kt` (4781 lines) are both god-files.** They centralize, respectively, *all* business logic and *all* voucher-screen UI+state. This is the natural consequence of an AI-assisted, prompt-by-prompt build process where new features get appended to the file that already "owns" the relevant concept rather than extracted into new files. Functionally fine today; will become a real velocity problem (merge conflicts, agent context-window pressure, slow comprehension) if the app keeps growing without a deliberate extraction pass (e.g., split `AppRepository` by domain: `VoucherRepository`, `PartyRepository`, `FinancialYearRepository`, etc., all sharing the same `AppDatabase`).
5. **`LedgerEntry.accountHead` and party/ledger-account matching by name string, not foreign key.** `closeFinancialYear()` and several queries match `entry.accountHead == "Party: ${party.name}"`. **Renaming a party will silently orphan all of its historical ledger entries from future FY-close calculations** (they'll no longer match the new name string and will stop contributing to the carried-forward balance). This is a real, currently-latent data-integrity bug waiting for a user to rename a party. If you're asked to "let users rename parties," you must also handle re-pointing or re-writing the `accountHead` string on every historical `LedgerEntry`, or switch the join to use `Party.id`/`LedgerAccount.id` instead of name.
6. **Full recalculation, not incremental, on every voucher save.** `recalculateOutstandings()`, `recalculateProductStocks(fy)`, and `updatePartyAnalytics(fy)` each re-read and re-derive state for the entire financial year on every single voucher save/delete. Fine at small-shop transaction volumes (the target user); **will become a real latency problem** once a shop accumulates thousands of vouchers in a year, since every single new sale re-walks the whole year's history three times. If performance complaints surface, this is the first place to profile, and the fix is incremental delta-updates rather than full recompute (nontrivial — would need careful handling of edit/delete cases that currently rely on "just recompute everything" being trivially correct).
7. **Two parallel reminder-schedule storage systems** (raw `SharedPreferences` vs. a Room-adjacent raw-SQL `reminder_schedules` table) — see §6.4 point 3. Consolidate onto one before adding new reminder features, or you will fix a bug in the wrong store.
8. **Hardcoded 15-day default credit term** in `saveAndPostVoucher`, even though `Party.creditDays` exists as a configurable field — confirm with product owner whether `Party.creditDays` is supposed to override this default (current code does not reference `party.creditDays` at all in the bill-receivable due-date calculation) and wire it through if so.
9. **Retrofit/OkHttp/Moshi/`play-services-location` dependencies present but no corresponding live network feature found in this pass.** Either dead weight (increasing APK size and ProGuard surface for no benefit) or scaffolding for an unfinished feature (GST portal verification? PIN-code geocoding via Play Services Location rather than a static PIN database?). Worth a direct question to the product owner before removing, since "looks unused" in a quick scan isn't proof of zero usage.
10. **Testing infrastructure (JUnit/Espresso/Robolectric/Roborazzi) is configured in `build.gradle.kts` but this scan found no substantial populated test suite.** Any "add tests" request should expect to be writing from near-zero, not extending existing coverage — and `exportSchema = false` (point in §5) blocks Room's standard migration-test helper until schema export is turned on.
11. **`my-upload-key.jks` fallback path in the release signing config.** Verify this keystore file is `.gitignore`'d and was never accidentally committed; if it's in git history at any point, the signing key must be treated as compromised and rotated before any Play Store submission.
12. **`.env.example` references a `GEMINI_API_KEY`** that appears to be a leftover from when this project may have been scaffolded/exported via Google AI Studio (`metadata.json`'s `majorCapabilities: ["MAJOR_CAPABILITY_SERVER_SIDE_GEMINI_API"]` is the same tell). Nothing in the current Kotlin code reads this key. Harmless to leave, but don't assume it wires up to a real Gemini integration in this app — it doesn't, in this snapshot.

## 9. Operational / deployment notes

- **Build:** `./gradlew assembleDebug` for a debug APK; release requires `KEYSTORE_PATH`/`STORE_PASSWORD`/`KEY_PASSWORD` env vars (or the fallback local keystore — see debt #11) plus `.env` populated from `.env.example` (currently only a vestigial Gemini key placeholder, so a fresh `.env` copy is sufficient for a normal build).
- **First-run UX contract:** splash → Setup screen (if no `BusinessProfile` row exists) → Dashboard. Any change to the onboarding flow must preserve this gate, since `AppViewModel.isSetupCompleted` is the single source of truth the entire nav shell branches on.
- **No backend / no sync.** This is a fully offline, single-device, local-SQLite app in this snapshot. There is no server component, no auth, no multi-device sync. Any feature request implying "sync across devices" or "cloud backup" is a net-new architecture decision, not an extension of existing code.
- **PDF generation depends on a `WebView`** (`configureInvoiceWebView`, `WebViewPdfWriter.java`) — this means invoice rendering quality is bounded by the device's WebView/Chromium version, and any HTML/CSS used in `buildInvoiceHtml` must stay within what a stock Android WebView reliably renders to a print/PDF surface (no exotic modern CSS without testing on a real low-end device, since the target user base skews toward budget Android phones).
- **Changelog/"What's New" dialog** is data-driven from `app/src/main/assets/changelog.json`, gated by `AppPreferences`' last-seen-version flag — when bumping `versionName`, also update `changelog.json` or returning users will see a stale or missing "what's new" dialog.

## 10. Quick-reference: "If you need to..." playbook for an agent

- **...add a new voucher type:** extend the `type` string set, add a branch in `AppRepository.saveAndPostVoucher`'s ledger-generation `when` block, decide its `shouldPostAccounts` eligibility, and check whether `recalculateProductStocks`'s movement-direction `when` block needs a new case (only if the type affects inventory).
- **...change GST calculation rules:** you must update the interstate/intrastate check in *every* screen listed in §6.1 (`VouchersScreen.kt`, `QuickSaleScreen.kt`, `ReportsScreen.kt`), or first extract a shared `GstCalculator` and migrate all call sites — flag this as a refactor-first task if asked to change GST logic in more than one place.
- **...fix or extend email reminders:** first determine whether the request is about the WorkManager+Intent-chooser path or the JavaMail-SMTP path (§6.4) — they are different code paths with different reliability/security profiles, and conflating them in a fix will likely solve the wrong half of the problem.
- **...add a Room column/table:** add to the `@Entity`, add a real `Migration`, **and** add/extend the matching `ensureColumn`/`ensure*Table` guard in `AppDatabase.kt`/`Migrations.kt` so existing installs self-heal on next open (§5).
- **...allow renaming a party or ledger account:** you must also handle re-pointing historical `LedgerEntry.accountHead` strings (§8 debt #5) or financial-year-close calculations will silently drop that history.
- **...improve voucher-save performance:** look at the three full-recalculation calls at the end of `saveAndPostVoucher` (§8 debt #6) before assuming the bottleneck is elsewhere (Compose recomposition, DB indexing, etc.).
- **...work on the KMP/multiplatform rebuild:** that code is not in this repository. Confirm with the user which checkout/branch holds it before attempting to "find" or "continue" that work here.
- **...harden security:** implement SQLCipher for `AppDatabase`, encrypt `BusinessProfile.smtpPassword` at rest (Android Keystore-backed), audit `backup_rules.xml`/`data_extraction_rules.xml` for DB exclusion, and move CSV/PDF exports off public storage — none of this exists yet in this snapshot (§7).
